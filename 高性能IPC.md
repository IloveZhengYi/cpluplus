# 架构

## [FastDDS]

🧠 分层结构

FastDDS ≈ RTPS协议栈 + DDS API

FastDDS从上至下分为三层：
 · DDS层：用户层API
 · RTPS层：通信协议层
 · Transport层：传输层封装

🧠 线程模型

· dds.ev.0 DDS定时线程（取决于创建了几个参与者实例）
· dds.udp.<multiple-port> DDS组播接收线程
· dds.udp.<meta-unicast-port> DDS匹配数据接收线程-单播
· dds.udp.<user-unicast-port> DDS用户数据接收线程-单播

· dds.log DDS 内部的异步日志输出线程
· dds.async.0.0 流控使用到的异步发送线程

Ps: 多线程模型能提升并发能力, 在资源受限和高负载情况下会引起调度抖动.

🧠 控制面 vs 数据面

· 控制面: metatraffic - 负责服务发现和匹配, 体现在PDP和EDP阶段
· 数据面: usertraffic - 实际业务数据的发送, 追求极致吞吐和确定性延迟

📌Note: 
数据面和控制面通过不同处理线程,和不同的底层传输层协议和地址来实现物理隔离.

❓ Q&A
Q.关于大规模情况下产生N^2的流量风暴
A.
 - Discovery Server模式解决了Simple模式的流量风暴问题, 控制面中心化, 数据面保持P2P直连, 实现了控制面和数据面路径的完美解耦.
 - 通过Static Discovery彻底关闭EDP阶段的控制面流量. 因为车载这种拓扑固定的场景, 我们不需要动态发现, 可以将节省出来的带宽资源给数据面, 同时规避控制面协议栈带来的CPU抖动. 安全性

Q.关于Discovery Server单点故障
只会影响后续的服务发现和匹配, 对数据面无影响.

🚀 FastDDS 架构主链

FastDDS架构本质是一个事件驱动的P2P异步系统, 整体可分为控制面和数据面, 遵循着服务发现, 匹配, 通信的闭环.
 1.控制面: 启动后通过SPDP组播发现参与者节点, 随后通过SEDP交换数据读写的Topic和QoS属性, 完成Endpoint的匹配.
 2.数据面: 在匹配完成后, 数据从应用层下沉到RTPS曾进行分片和序列化, 最后交由Transport层进行物理投递.
 3.控制面负责发现和匹配, 数据面负责数据传输.
 
🔴核心问题: 

· Simple模式的流量风暴问题
-> 解决方案: Discovery Server(架构快速迭代, 灵活性) 和 Static Discovery(拓扑固定 + 安全性 + 无EDP流量)
-> 当前选择: Discovery Server
-> 引入的问题:
 · 不同域单独重启, Tbox没有监测到Server断开的引入了半连接以及资源泄漏问题.

· 回调阻塞问题
FastDDS 直接在接收线程中执行 on_data_availble 回调, 如果业务处理阻塞, 会影响后续数据处理, 甚至2.6版本会导致匹配稳定性.

· 周期性发送组播流量
参与者数量众多且网络受限架构中会打满网络带宽

## [IceOryx]

🧠 概述
IceOryx是基于共享内存的零拷贝通信框架, "中心化管理，去中心化传输"的架构, 控制面和数据面分离. 
 - 控制面: RouDi负责资源管理与注册.
 - 数据面: 共享内存实现零拷贝.

🚀 IceOryx 架构主链
1.RouDi负责创建并初始化共享内存.
2.进程通过UDS注册, 拿到共享内存和内存池相关信息.
3.发布者 loan 借用内存并写入数据, 将chunk描述投递到接收方的接收队列
4.接收方直接访问内存, 无需拷贝.

⚡ 零拷贝机制
背景: 自动驾驶/机器人场景下, 数据量大, 频率高, 资源受限, 实时性要求高.
传统IPC存在较大的CPU开销和延迟抖动:
 - 内存拷贝次数: 发送端用户态 -> 内核 -> 接收方用户态 2 次
 - 上下文切换次数: 2次系统调用
IceOryx 通过共享内存 + loan + 指针传递, 实现数据传输不经过内核, 不发生拷贝.

内存池: 使用预分配的多级内存池管理, 避免动态分配带来的延迟和碎片问题, 同时提升确定性.
相对指针: 共享内存在不同进程的虚拟地址空间中, 映射的起始地址通常是不一样的. IceOryx 通过 RelativePointer 替代绝对地址 (id + offset)

🔴 核心问题
Q.内存池耗尽
A.配置最大可持有的chunk上限, 以及丢弃策略.

Q.优先级反转 - 低优先级消费者占用内存块, 导致高优先级生产者被阻塞
A.独立内存池 (物理隔离), 内存限制.
Ps: 跨进程的共享锁使用 PTHREAD_PRIO_INHERIT 使内核暂时提升后者的优先级, 使其尽快完成任务并释放锁.

Q.如果是高优先级任务是控制指令, 每一帧都不能丢, 此时内存满了, 如何做?
A.控制指令通常很小, 预留100%冗余的独立内存池, 牺牲内存利用率换取100%确定性.

Q.RouDi单点故障
A.RouDi崩溃不会影响现有的通信链路, 但新成员无法加入, 并且IceOryx并没有考虑RouDi崩溃的问题.

❓ Q&A

Q.上下文切换的开销
A.CPU寄存器、栈相关、虚拟内存相关. 这个开销是几百纳秒或微妙级别, 最大开销是缓存污染: flush TLB 和 Cache Miss
-> 优化方案: 可以通过协程, 减少线程数, 绑核, 使用自旋或忙等来解决.
-> 线程切换vs进程切换: 线程切换不换CR3, 不会导致 flush TLB, 开销通常是 1/3 - 1/5
A.用户态到内核态切换严格来说并不算上下文切换, 只会保存用户寄存器到内核栈, 开销小.

Q.内存池是怎么管理的? 高并发申请内存会锁死吗?
A.内存池是RouDi启动时创建并切分成多级内存池, 使用无锁队列进行管理, 高并发不会出现互斥锁导致的上下文切换.

⚖️ Diff: FastDDS vs IceOryx
FastDDS主打分布式 + 网络通信 + 高灵活性
IceOryx是单机、零拷贝、机制低延时.

🔍 Ext: 除了 IceOryx 还有哪些可以彻底绕过内核的通信技术
DPDK和RDMA暂略, 与高性能网络编程联动
